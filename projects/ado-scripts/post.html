Azure DevOps

Automated Scripts

Summary

During my rotation into the Release Management team at General Motors, I had the opportunity to work with Azure DevOps (formerly Team Foundation Server). ADO is a very powerful platform but it was very underutilized due to the corporate policies that prevented the use of open source plugins/extensions.

I worked on a suite of scripts that improved efficiency, development team flow, and automated tasks. These scripts included:

  • Build Info: Manually passed build pipeline variables via JSON file through the drop folder. This was done manually since GM corporate policy prevented the use of plugins that would do this for us.
  • JKS Certificate Expiration: Build pipeline script that failed builds and notified dev teams (scheduled on a weekly cadence) if any dev/prod JKS files were expired.
  • Branch Merge Automation: Release pipeline script that merged Release branches into Prod/Master and back into Develop after the successful push to 3 production environment servers. This saved a lot of time for development team leads and promoted proper git flow/hygiene.
  • Git Release Branch Hygiene: As part of the Branch Merge Automation scripts- an additional script was created as a monitoring tool when release branches had commits pushed into them. This metric dashboard was created because proper git hygiene was not followed so teams were able to monitor which project and which branch was not following proper git flow.
  • Blue/Green Deployment: These were suite of release pipeline scripts that would perform the blue/green deployments into the PCF spaces (specified in project manifest.yml files). More so- these scripts were overhauled as the previous ones were hardcoded and did not allow us to scale easily.
  • PCF Redeployment Automation: As part of maintaining the PCF (Pivotal Cloud Foundary) space, redeployments were required within 30-45 days from release. This script would push the latest release/build numbers onto a markdown page to be used for data entry for our redeployments.
  • Redeployment Verification: Part of releasing enterprise applications is validating that the applications we pushed to production servers were operating properly. This script hit every health actuator endpoint on every microservice and notified which apps were online/offline. This was created so that the release management team would no longer need to wait on dev team leads to verify.
